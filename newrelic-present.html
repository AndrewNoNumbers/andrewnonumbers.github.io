New Relic

Increasing feedback and fixing navigation flaws on New Relic docs

What is New Relic?

New Relic is a full-stack monitoring platform, used by half of the Fortune 100. Software engineers use New Relic to troubleshoot software performance issues and outages.

Problem exploration and convergence

The kick off to any design project is to understand what problem I'm solving and why it matters. Sometimes, the problem surfaces organically through continuous discovery; other times, the problem is given to design because someone internally believes there's a problem.

For this project, it was the latter: the Docs team brought up that they believed the Docs site navigation needed a redesign, and would appreciate design help.

It's not a bad place to be when stakeholders proactively invite design to redesign something. However, it was up to me as the designer assigned to figure out exactly what pain points users experience with the Docs site navigation, so that we could test and ensure that our redesigned navigation actually works better.

I partnered with a UX researcher at New Relic to run and analyze a UX survey on the Docs site, and followed up with a series of observational interviews of actual New Relic users using the Docs site. What we found was that not only did 'Ease of use' and 'Ease of navigation' score uncommonly low on the survey, real users had a shared set of struggles with getting around Docs.

These were the struggles I observed

Each section has its own navigation

Clicking on a top-level topic area of Docs leads the user to a new navigation. Users exhibited uncertainty about how the new navigation relates to the navigation that were looking at a second before, and most took a while to recall how to get back to the main navigation.

Back is broken

I found that back navigation is broken around 5% of the time. When the user clicks back in the browser, the page would try to go back but instantly go forward again.

Shortcut pages are jarring

Shortcut pages are pages that don't live where it looks like it lives. In this example, "APM best pracitces guide" is not an article that lives in the "Monitor your app's performance" section even though it appears like it, but rather it links to an article in the "Guides and solutions" section. Shortcut pages have unexpected behavior and confuse users by taking them somewhere they had no expectation of going to, and causing them to lose their place.

Top-level categories are unclear

Top-level categories are not easily recognizable to users. The top-level navigation presented the vast set of New Relic's capabilities in an alphabetical list. I found that many users did not understand most of them, because they use only a small set of New Relic's capabilities, so they end up looking specifically for the thing they recognize.

Solution exploration and convergence

With a stronger understanding of the problems plaguing the current navigation experience, I set forth to figure out how a navigation redesign could alleviate the issues.

As with any product, but especially navigation, the ideal design would consider how these problems could be solved holistically, rather than piecemeal.

  • See how I moved Docs to a design system

    Before getting to the UX of solutions, I found that the Docs site was in an unusual space of not having a design system or guidelines. This was the effect of having a decentralized design organization — the marketing team had a design system, and the product team had a design system. However, other teams like the Docs team which lived under neither organization fell through the cracks and did not have awareness or a working relationship with either design system.

    Building without a design system means there's no documentation of design patterns and reuse, and creates risk of introducing inconsistencies and redundancies when future designers work in the same area.

    In order to prevent this and to set the Docs team up for stronger design consistency in the future, I evaluated both the brand design system and the product design system. I then set up meetings with the Head of Brand Design and the Design Systems Lead on the product side to build alignment on where I should guide the Docs site.

    The alignment we built was that the Docs site should follow the marketing design system, since that system is more fitting for a content-heavy site like Docs, and also because Docs should fit into the family of external-facing sites, so a consistent styling makes sense.

    Because the Docs team was looking for a redesign of site navigation, this was the perfect opportunity to set the site up with an aligned design system.

Addressing each issue

A navigation system that doesn't require mental gymnastics

This was the crux of what many internally believed to be the cause of navigation issues. Clicking on any topic area from the main navigation would open up a different navigation that appears completely different from the main navigation.

While engineering was exploring the feasibility of combining the disparate sections into a unified navigation, I took the time to mock up how we could resolve some confusion if a unified navigation doesn't work out.

While the addition of the 'Back to home' and '24 articles in this section' would be low lift for engineering, the ability to have breadcrumbs runs into significant challenges.

Luckily, the engineering team had found it quite feasible to build a unified navigation, and the team had a lot of momentum toward it, so that's what I explored next.

A unified nav has many benefits, such as easier exploration into different directories. It also reduces the cognitive load when changing between different topic areas.

However, there were also interaction design challenges, especially since New Relic had never built a tree-view navigation.

Back no longer breaks

Back being broken turned out to not be a Docs-specific issue, but rather a newrelic.com-wide analytics script that broke the ability for users to go back 5% of the time. When the users presses back in the browser or swipes back on a Mac or mobile device, the site would reload the current page.

It was uncommon enough that no other internal stakeholder had noticed it or brought it up, but was affecting all of our sites including newrelic.com, our forums, our docs site, and other sites.

I take credit only for surfacing the issue and funneling it to the right team that could figure out what was wrong and fix the loading of the script across all the sites. The responsible engineering team was the true hero in resolving the issue.

Shortcut pages get an icon

In the old navigation, not only was there no indicator that a shortcut page was any different than other pages, it would take users to a totally different navigation that they had no expectation of going to.

With the redesigned unified navigation, shortcut pages have a shortcut icon, and users are scrolled to the part of the navigation the shortcut page points to.

Top-level categories are grouped semantically

I accomplished this by leading the Docs writing team in an affinity mapping exercise, where each writer was able to do their own grouping, and then we came together to see how each person's groupings compared to others' groupings. This led to a natural alignment on how different Docs topic areas relate to each other, and how they should be labelled.

Getting the solution shovel-ready

Shovel-ready is a term one of my design colleagues uses to communicate that a design project is ready for engineers to start building. Even after the UX changes had been tested and gained stakeholder approval, there were some final details to figure out.

A project isn't done just because it's shovel-ready and engineers are working on it. As things are built, engineering will often have questions about details, and sometimes project owners need to make hard decisions in cutting some scope, in which case the designer's input is important to make sure the built experience is still holistically useful.

In this case, because all the changes were front-end, the engineering team was able to fit them all into one sprint, and no scope needed to be cut.

Having tested the individual changes to the navigation, we had high confidence that it would solve users' frustration with navigating Docs. However, as a final check, I ran a round of holistic usability testing comparing the newly-built Docs navigation with the existing Docs navigation.
